01

Data & preprocessing

fraud_preprocessing.py

The transaction and identity tables are left-joined on TransactionID — not every transaction has a matching identity record, so a has_identity flag marks the ones that do. Identity columns are named inconsistently across files (id_01 in train, id-01 in test); those get normalized before anything downstream touches them. Known categoricals (ProductCD, card1–card6, addr1/2, both email-domain columns, M1–M9, DeviceType, DeviceInfo, id_12–id_38) are cast explicitly so nothing downstream treats them as numeric by accident.

A fitted NumericPreprocessor then turns everything into one consistent model matrix: booleans map to 0/1, numeric gaps are median-imputed (with a missingness indicator kept alongside), low-cardinality categories are one-hot encoded, and high-cardinality ones are frequency-encoded instead. The fitted plan is pickled once and reapplied identically to validation and test data, so column shape never drifts between splits.

Exploration

fraud_exploration.py

A dataset-wide audit flags missingness, near-constant columns, likely identifiers, and duplicate columns. Every feature gets a signed-AUC association score against isFraud, and a separate check tests whether a column's absence — not its value — is itself predictive. Standardized logistic coefficients and an LDA projection give a linear read on which features separate the two classes. A stratified sample of confirmed-fraud rows is then pairwise-compared (numeric closeness plus categorical exact-match) into a similarity matrix.

Feature analysis & selection

fraud_feature_analysis.py

Everything here is leakage-aware: features are ranked on a train-only split, then checked for stability against a held-out split so the ranking can't quietly borrow signal it won't have at inference time. Redundant/correlated features are grouped into clusters, model-based importance comes from a standardized logistic fit, and a class-specific permutation test measures how much each feature's removal hurts. All four signals combine into one composite score — 45% predictive power, 35% model importance, 20% permutation importance — penalized for being near-constant, identifier-like, unstable, or redundant.

Model training & comparison

fraud_model_training.py

Five classifiers — all built from scratch, no sklearn/PyTorch in the models themselves — trained on the same stratified split (442,905 train / 147,635 validation rows) using the top 60 features: a precision-weighted rules-based scorer, a Gini-split decision tree with human-readable leaf paths, a bootstrap-bagged random forest of those same trees, a one-hidden-layer neural network with manual backprop, and a Beta-noise random baseline centered on the true fraud rate as a sanity floor.

06

Verdict

Random forest is the model to build on

It leads on both ROC-AUC (0.811) and, more importantly given the 3.5% fraud rate, PR-AUC (0.293 — nearly double the neural network's). The rules-based scorer is barely above chance (AUC 0.505) despite firing at 96% precision on the rare occasions it does trigger — useful as a supplementary hard rule, not viable as a primary classifier. The random-contender baseline confirms every learned model is picking up real signal, not noise.

All four learned models sit at high recall / low precision at the default 0.5 threshold — expected for this class imbalance, but not deployment-ready as-is.

  1. Tune the decision threshold on random forest instead of using 0.5 — precision is currently 11%, which would flood a review queue.
  2. Re-train on the 25-feature subset directly in the model-training stage; it already matches the 248-feature set on held-out AUC with a fraction of the inputs.
  3. Calibrate probabilities before using random forest output as a risk score rather than a binary flag.
